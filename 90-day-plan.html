Entry plan

Read the system. Establish continuity. Earn scale.

Day-90 objective

Produce one evidence-backed production improvement and a reusable Enterprise AI Backplane practice without pretending the undisclosed client environment is already known.

Operating assumptions to verify immediately

Role center of gravity

Confirm the balance among model development, ML platform engineering, enterprise integration, technical leadership, and customer-facing solution work.

Current production estate

Identify deployed systems, clouds, frameworks, pipelines, APIs, data sources, evaluation approaches, monitoring, and incident history.

Decision rights

Clarify architecture authority, release approval, risk escalation, production ownership, support ownership, and retirement authority.

Success measures

Agree which workflow outcomes, reliability signals, costs, controls, adoption evidence, and learning measures matter.

Principles

Work backward from consequence

Define the workflow, user, baseline, risk, and decision before architecture.

Fix the weakest interface

Do not optimize the model while context, runtime, authority, or evidence remains the system limit.

Harvest every proof

Each release should leave reusable architecture, tests, evidence, or a fast no.

Days 0-30

Trace the production reality.

Portfolio and use-case map

Document active, piloting, queued, paused, and retired systems; users; workflows; outcomes; sponsors; risk class; and technical owners.

Architecture and dependency map

Trace models, prompts, retrieval, data stores, APIs, services, queues, cloud resources, authentication, security controls, and downstream actions.

Evaluation and telemetry map

Inspect offline evaluations, production metrics, task evidence, human feedback, drift, incidents, cost visibility, and alert ownership.

Authority and operating map

Identify approvals, overrides, exception handling, escalation, release gates, rollback authority, support, and change cadence.

Days 31-60

Close the weakest interface.

Select a bounded proof

Choose one meaningful workflow whose friction, consequence, ownership, and technical boundary are visible enough to measure.

Write the six contracts

Make outcome, context, model, runtime, authority, and evidence explicit.

Harden one fault path

Add the right test, monitoring, fallback, timeout, escalation, approval, rollback, or documentation.

Define acceptance

Specify functional, quality, reliability, cost, control, adoption, and recovery evidence.

Run a production drill

Exercise context drift, dependency failure, authority conflict, or quality decay.

Prepare the decision

State what evidence will support scale, further bounding, hold, rollback, or stop.

Days 61-90

Turn the proof into engineering leverage.

Observe the bounded release

Track output quality, task outcome, latency, availability, cost, user behavior, overrides, incidents, and recovery.

Run the learning loop

Review false confidence, failure modes, support burden, user feedback, control exceptions, and ownership friction.

Codify reusable interfaces

Create architecture notes, evaluation templates, fault tests, control patterns, telemetry expectations, documentation, and review criteria.

Recommend portfolio disposition

Scale, further bound, hold, roll back, or retire. State evidence, unresolved risk, economics, ownership, and the next decision date.

Risks to surface early

Role mismatch

The position may be primarily novel model research.

Access without authority

A staff contributor may see risk but lack decision rights to close it.

Metric substitution

Offline scores, uptime, or usage can obscure poor workflow outcomes.

One-off success

A proof may work without producing reusable architecture or practice.

Ownerless handoff

Technical delivery can precede operating ownership.

Scale before recovery

Reach should not expand faster than monitoring, rollback, and incident learning.
